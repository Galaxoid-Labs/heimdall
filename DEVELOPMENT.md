## Strategy: webview/webview first, native backends later

We bootstrap on **[webview/webview](https://github.com/webview/webview)** (the
tiny MIT C/C++ library wrapping WKWebView / WebKitGTK / WebView2) rather than
hand-writing three native backends up front. Reason: its C API already gives us
most of the bridge.

- `webview_bind(name, fn)` + `webview_return(seq, status, result)` = the entire
  **request/response** path. JS calls `window.<name>(args)` and gets a Promise;
  our callback receives `(seq, req_json)`; we resolve with `webview_return`. No
  hand-rolled postMessage/pending-map needed on this path.
- `webview_init(js)` injects our `__HEIMDALL__` shim before page load.
- `webview_eval(js)` pushes **events** (Odin → JS).
- `webview_dispatch(fn, arg)` is `dispatch_main` — one of only two calls safe from
  a background thread (the other is `webview_terminate`).

What we give up by starting here (deferred to the native-backend phase): native
menus, tray, and a custom `app://` scheme handler. The scheme gap is the only one
that bites early — see Phase 3 for the localhost-server workaround.

**Decision:** this resolves CLAUDE.md open-question #5 — bootstrap on
webview/webview, keep the internal backend vtable so native backends can slot in
later without touching the bridge or user code.

## Package & layout rules (read before scaffolding)

- **No `src/` package folder.** In Odin the directory name *is* the import name,
  and `import "src"` is meaningless. Name packages for what they are.
- The framework package is `heimdall` (folder `heimdall/`). The webview bindings
  are their own package `webview` (folder `heimdall/webview/`), imported by the
  framework as a subpackage.
- **Vendored by default.** A generated app carries its own copy of the framework.
  No package manager in Odin, so this is the idiomatic, portable choice —
  `git clone myapp && heimdall build` works with only the CLI installed.

**Framework repo:**

```
heimdall/
  heimdall/            # package heimdall (the importable library)
    app.odin
    service.odin
    bridge.odin
    events.odin
    bridge_js.odin
    schema.odin
    assets.odin
    server.odin        # tiny localhost static server (prod asset serving)
    threading.odin
    errors.odin
    webview/           # package webview — Odin bindings to webview/webview
      webview.odin
      lib/             # vendored upstream: webview.h (+ built libwebview per OS)
  cli/                 # package main → the `heimdall` binary
    main.odin
    cmd_*.odin
    templates/
  examples/
    hello/
    filebrowser/
  CLAUDE.md
  DEVELOPMENT.md       # this file
```

**Generated user app** (default = vendored, root main package, no `src`):

```
myapp/
  main.odin            # package main;  import hd "heimdall"
  services.odin        # package main
  assets_gen.odin      # generated by `embed` (gitignored)
  heimdall/            # vendored framework (the import target; subdir of root pkg)
  web/                 # frontend (any stack) → web/dist
  heimdall.toml
  build.sh
```

`import hd "heimdall"` resolves to the `./heimdall/` subdirectory of the root
package — no collection flag, no `..`, no `src`. For larger apps that want the
backend in its own named folder, use a collection instead
(`odin build app -collection:deps=deps`, `import hd "deps:heimdall"`); the CLI
injects the flag. Default template stays collection-free for simplicity.

### webview/webview bindings sketch (`heimdall/webview/webview.odin`)

```odin
package webview

import "core:c"

when ODIN_OS == .Windows {
	foreign import lib { "lib/webview.lib" }
} else when ODIN_OS == .Darwin {
	foreign import lib { "lib/libwebview.a", "system:c++" }
} else {
	foreign import lib { "lib/libwebview.a", "system:stdc++" }
}

Webview :: distinct rawptr   // webview_t (opaque)

Hint :: enum c.int { None = 0, Min = 1, Max = 2, Fixed = 3 }

Native_Handle_Kind :: enum c.int { UI_Window = 0, UI_Widget = 1, Browser_Controller = 2 }

Dispatch_Proc :: #type proc "c" (w: Webview, arg: rawptr)
Bind_Proc     :: #type proc "c" (seq: cstring, req: cstring, arg: rawptr)

@(default_calling_convention="c", link_prefix="webview_")
foreign lib {
	create             :: proc(debug: c.int, window: rawptr) -> Webview ---
	destroy            :: proc(w: Webview) ---
	run                :: proc(w: Webview) ---
	terminate          :: proc(w: Webview) ---                                  // thread-safe
	dispatch           :: proc(w: Webview, fn: Dispatch_Proc, arg: rawptr) ---  // thread-safe
	get_window         :: proc(w: Webview) -> rawptr ---
	get_native_handle  :: proc(w: Webview, kind: Native_Handle_Kind) -> rawptr ---
	set_title          :: proc(w: Webview, title: cstring) ---
	set_size           :: proc(w: Webview, width, height: c.int, hints: Hint) ---
	navigate           :: proc(w: Webview, url: cstring) ---
	set_html           :: proc(w: Webview, html: cstring) ---
	init               :: proc(w: Webview, js: cstring) ---
	eval               :: proc(w: Webview, js: cstring) ---
	bind               :: proc(w: Webview, name: cstring, fn: Bind_Proc, arg: rawptr) ---
	unbind             :: proc(w: Webview, name: cstring) ---
}

// `return` is a keyword, so bind webview_return explicitly:
@(default_calling_convention="c")
foreign lib {
	@(link_name="webview_return")
	resolve :: proc(w: Webview, seq: cstring, status: c.int, result: cstring) ---
}
```

> Callbacks crossing the FFI boundary MUST be `proc "c"` and must set up an Odin
> `context` (use `context = runtime.default_context()` or a stored app context at
> the top of each callback) before calling anything that touches allocators.

### Vendoring webview/webview

Upstream is C++ (`webview.h` declarations + a `webview.cc` you compile once into
`libwebview.a`). Vendor the header + a built static lib per OS under
`heimdall/webview/lib/`. Link deps:

- **Linux:** `libwebview.a` + `-lstdc++` + WebKitGTK (`pkg-config --libs webkit2gtk-4.1`).
- **macOS:** `libwebview.a` + `-lc++` + `-framework WebKit -framework Cocoa`.
- **Windows:** `webview.lib` + WebView2 loader; Evergreen runtime present on
  current Windows (verify in `doctor`).

`heimdall doctor` checks these are installed and prints fixes.

## Phase 1 — The bridge: services & commands over `webview_bind`

**Goal:** `invoke("greeting.greet", {name})` round-trips to an Odin handler.

- [ ] `service.odin`: `service()` + parametric `command()` registration (per
      CLAUDE.md contract); registry = `map[string]Thunk` keyed `"service.command"`.
- [ ] Single `webview_bind("__heimdall_invoke", ...)` C callback that: parses
      `{service, command, args}` from `req`, looks up the thunk, runs it,
      `json.marshal`s the result, calls `webview.resolve(seq, 0/1, json)`.
- [ ] `bridge_js.odin`: the `__HEIMDALL__` shim string, injected via
      `webview_init`. Provides `invoke(name, args)` wrapping the bound function.
- [ ] Thunk: `raw_json → json.unmarshal(A) → handler(state, A) → json.marshal(R)`.

**Done when:** the `hello` example calls a command from a button and renders the
returned string; an Odin-side error rejects the JS Promise.

## Phase 2 — Events + threading

**Goal:** Odin can push `emit("file.progress", payload)` to a JS `on(...)` handler.

- [ ] `events.odin`: `emit(app, name, payload)` → marshal → `webview_eval`
      (`__HEIMDALL__._event(name, payload)`).
- [ ] Shim: `on(name, handler)` registry + `_event` fan-out; return unsubscribe.
- [ ] `threading.odin`: `dispatch_main` wraps `webview_dispatch`. Rule: any
      `eval`/`resolve`/`emit` from a worker thread hops through `dispatch_main`.
- [ ] Document the UI-thread rule loudly in code comments.

**Done when:** a command spawns a worker that emits progress events the UI shows
live, with no main-thread violations.

---

## Phase 3 — Assets: embed + serve

**Goal:** prod app loads its embedded SPA; dev loads the bundler's HMR server.

- [ ] `embed` step: `dist/` tree → `assets_gen.odin` (`map[string]Asset` with
      `#load` + MIME).
- [ ] `server.odin`: tiny localhost static server (`core:net`) serving the embedded
      asset map on `127.0.0.1:<random free port>`. (webview/webview has no custom
      scheme handler, so we serve over loopback instead of `app://`.)
- [ ] URL switch: dev → `dev_url` (e.g. `http://localhost:5173`); prod → the
      loopback server. Gate via `-define:HEIMDALL_DEV=true`.

**Done when:** `heimdall build` produces a single binary that opens the real SPA
with working `invoke`; `heimdall dev` shows HMR.

> The custom `app://` scheme is deferred to Phase 7 (native backends). Loopback is
> the pragmatic webview/webview answer and is fine for MVP.

## Phase 5 — CLI

**Goal:** `heimdall new → dev → build` works end to end.

- [x] `new <name>`: scaffold (vendor `heimdall/` in, `--frontend vanilla|sveltekit`,
      `--pm bun|npm|pnpm|yarn|deno`).
- [ ] `dev`: start frontend dev server, build app with `HEIMDALL_DEV=true`, launch
      at dev URL, rebuild+relaunch on change.
- [ ] `build`: frontend build → `embed` → `odin build -o:speed` → binary.
- [ ] `embed <dir> <out>`: standalone asset embed.
- [ ] `doctor`: verify `odin`, bundler, platform webview link deps; print fixes.

**Done when:** clone-template → `heimdall dev` opens a working window with a live
`invoke` round-trip in under a couple of minutes.

## Phase 6 — Typed bindings (`generate-bindings`)

- [ ] Schema-dump mode (`-define:HEIMDALL_SCHEMA=true`): init services, walk the

      (`struct_field_names/types/tags`), emit a JSON schema, exit.

autocomplete for every command; deleting it doesn't break the build.

## Phase 7 — Native backends

**Goal:** drop the webview/webview dependency where we need more than it offers.

- [x] Internal backend vtable (`backend.odin`) — the seam every backend implements.
- [x] **macOS** WKWebView (objc interop) — `backend_darwin.odin`, the DEFAULT on
      macOS. Custom `app://` scheme handler, native menus, `should_quit`.
- [x] **Linux** GTK4 + libadwaita + webkitgtk-6.0 (`foreign import` C / GObject) —
      `backend_linux.odin`, the DEFAULT on Linux. AdwHeaderBar title bar that follows
      the system light/dark theme, custom `app://` scheme, `GtkPopoverMenuBar` menus,
      `should_quit` via `close-request`. All `_probe*` pass identically to macOS.
- [ ] **Windows** WebView2 (COM) — `backend_windows.odin`. **← NEXT.** Last (most
      tedious). See the Windows section of `docs/platform_notes.md`.
- [ ] Tray (reuse `tray-odin`) + native dialogs.
- [x] Bridge/services/events APIs unchanged across backends — proven on macOS
      and Linux (all `_probe*` pass identically on webview + native).

**Done (macOS + Linux):** `examples/hello` runs against the native backend with
menus + `app://` and identical user-facing code. Repeat the bar for Windows.

## Cross-cutting checklist

- [ ] Every public proc returning an `Error`/handle is `@(require_results)`.
- [ ] No `eval`/`resolve`/`emit` off the UI thread without `dispatch_main`.
- [ ] FFI callbacks set up an Odin `context` before touching allocators.
- [ ] `core:os` (redesigned) — explicit allocators, `^os.File`, `os.Error`.
- [ ] No methods; procedures over data; `create`/`destroy` naming.
