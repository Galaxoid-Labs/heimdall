# Heimdall

Heimdall builds native desktop apps from an **Odin backend** + a **web frontend**,
bridged together and embedded into a single binary (think Tauri/Electron, but
small and Odin all the way down). The framework is vendored into each app as
`./heimdall/` and imported as `import hd "heimdall"`.

For Odin syntax/idioms, use the **odin-lang** skill. This skill is the Heimdall
API + workflow.

## Three layers, one boundary

```
WEB (your frontend)   invoke("svc.cmd", args)  ──►  BRIDGE (Odin)  ──►  NATIVE SHELL
  window.heimdall     on("event", handler)      ◄──  your procs        (window + webview)
```

The web layer **cannot touch the OS** — everything privileged goes through a
**command** you register in Odin. Keep the frontend trusted; that boundary is the
security model.

## Commands (request/response)

A **service** is a struct holding state; a **command** is a proc over it taking
and returning JSON-marshalable Odin types. Marshalling is generated at compile
time from the types — no macros, no reflection.

```odin
// services.odin
Greeting     :: struct { prefix: string }       // service state
Greet_Args   :: struct { name: string }         // input from JS
Greet_Result :: struct { message: string }      // result to JS

greet :: proc(s: ^Greeting, args: Greet_Args) -> (Greet_Result, hd.Error) {
    return {message = fmt.tprintf("%s%s", s.prefix, args.name)}, nil
}
```
```odin
// main.odin
g := hd.service(app, "greeting", &greeting)   // namespace
hd.command(g, "greet", greet)                 // -> callable as "greeting.greet"
```

Returning a non-nil `hd.Error` **rejects** the JS promise. Commands run on the UI
thread; for off-thread work hop back via `hd.dispatch_main` (or just `hd.emit`,
which is already thread-safe).

## Events (push)

```odin
Progress :: struct { read, total: int }
hd.emit(app, "file.progress", Progress{read = 512, total = 1000})  // any thread
```

Declare a payload type for typed bindings (optional, additive):
```odin
hd.event(app, "file.progress", Progress)   // -> typed on("file.progress", ...)
```

## The JS side

**Recommended: the generated typed client** (`heimdall.gen.js`, regenerated by
`new`/`dev`/`build`). One object per service, typed args + results, typed events:

```js
import { greeting, on } from "./heimdall.gen.js"
const { message } = await greeting.greet({ name: "Jake" })
const off = on("file.progress", p => updateBar(p.read / p.total))
```

**Escape hatch (no build step):** `window.heimdall` is the runtime global —
`window.heimdall.invoke("greeting.greet", { name })` and `window.heimdall.on(...)`.
(There is no `__HEIMDALL__` alias; it was removed.)

## App_Config (passed to `hd.create`)

```odin
hd.create(hd.App_Config{
    title = "My App", width = 900, height = 600, resizable = true,
    dev_url = "http://localhost:5173",   // used by `heimdall dev`
    assets  = ASSETS,                    // embedded by `heimdall build` (from `embed`)
    icon    = #load("icon.png"),
    devtools = .Auto,                    // .Auto (dev on/release off) | .On | .Off
    menu = { ... },                      // native menu bar (see Menus)
    url_schemes = {"myapp"},             // deep-link schemes (see Deep linking)
    on_open_url = proc(app: ^hd.App, url: string) { ... },
    on_startup  = proc(app: ^hd.App) -> hd.Error { return nil },
    on_shutdown = proc(app: ^hd.App) {},
    should_quit = proc(app: ^hd.App) -> bool { return true }, // false vetoes a close
    // initial state: min_width/min_height, maximized, fullscreen, always_on_top, center, hidden
})
```

## Window control

`hd.window_minimize/maximize/unmaximize/show/hide/focus/center/close(app)`,
`hd.window_set_fullscreen(app, on)`, `hd.window_set_title/size`. Also exposed to
JS via the built-in **`win`** service: `win.minimize()`, `win.close()`, etc. The
name `win` is reserved.

## Paths (per-app dirs)

One cross-platform API for the OS's config/data/cache/log locations, namespaced by
`app_id` and created on first access. Returned strings are caller-owned.

```odin
cfg := hd.config_dir(app)                          // also data_dir/cache_dir/log_dir
db  := hd.app_path(app, .Data, "db/store.sqlite")  // builds the file path + makes parents
// hd.app_dir(app, kind) is the general form; Path_Kind :: enum {Config, Data, Cache, Log}
```

Set `app_id = "com.example.myapp"` in `App_Config` (falls back to a sanitized
`title`). Built-in **`paths`** JS service: `await paths.config()` → `{path}` (also
`data`/`cache`/`log`). `paths` is a **reserved** service name (like `win`). The
frontend has no OS access — use a path to display, or pass it to a command that
does the file I/O in Odin. See `docs/guide/paths.md`.

## Menus (native)

```odin
menu = {
    {label = "File", submenu = {
        {label = "New",   id = "file.new",  accelerator = "CmdOrCtrl+N"},
        {separator = true},
        {label = "Close", role = .Quit},
    }},
}
```
Items with an `id` emit a `"menu"` event `{ id }` (subscribe with `on("menu", …)`);
`role` items are standard actions. Use `CmdOrCtrl+…` for cross-platform accelerators.

## Deep linking (`myapp://`)

Set `url_schemes` (runtime) + `[bundle].schemes` in heimdall.toml (OS
registration). Incoming URLs arrive via the `on_open_url` hook and an `"open-url"`
event. macOS is fully supported; Windows/Linux handle cold-start today.

## CLI

| Command | Does |
| --- | --- |
| `heimdall new <name>` | Scaffold (`--frontend vanilla\|sveltekit`, `--pm …`). |
| `heimdall dev` | Frontend dev server + app; rebuild on change. (`--webview` opts out of native.) |
| `heimdall build` | Frontend build → embed → compile a release binary. |
| `heimdall bundle` | Package: macOS `.app`, Linux `.deb`/`.rpm`, Windows installer. `--sign`/`--adhoc`/`--notarize`. |
| `heimdall generate-bindings` | Emit the typed `.d.ts` client (commands + events). |
| `heimdall doctor` | Check toolchain (odin, bun, platform webview deps). |

Build defines (passed to `odin build`): `-define:HEIMDALL_DEV=true` (point at
`dev_url`), `-define:HEIMDALL_WEBVIEW=true` (force the webview/webview backend on
macOS), `-define:HEIMDALL_SCHEMA=true` (schema-dump mode).

## heimdall.toml

Optional. Top-level: `name`, `web_dir`, `dist_dir`, `dev_cmd`, `build_cmd`,
`dev_url`. Packaging under `[bundle]` / `[sign]`, with per-platform overrides
(`[bundle.macos]`, `[bundle.windows]`, …) resolved platform-first. Keep it small.

## Gotchas

- **Generated map-literal files** (`assets_gen.odin`) need `#+feature dynamic-literals`.
- **Odin filename suffixes** like `_js`/`_darwin`/`_windows` are build-target gates
  — never name a non-platform file `*_js.odin` (silently excluded elsewhere).
- **`fmt` treats `{` as a token** — don't pass text with literal braces (struct
  literals, JSON) as a format arg; use token replacement or `strings.write_string`.
- **Worker threads:** never `eval`/resolve/emit off the UI thread without
  `hd.dispatch_main` (`emit` already hops for you).
